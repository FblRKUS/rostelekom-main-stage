# CodeLens RAG

Система семантического поиска по кодовой базе (RAG) с использованием Python AST, ChromaDB и Streamlit.

## Архитектура
Система состоит из 4 логических модулей:
1. **Парсер и Чанкер (`parser.py`)**: Использует модуль `ast` для парсинга Python кода. Чанком считается строго одна функция или один класс. В качестве бонуса методы классов извлекаются как отдельные чанки с именем `ClassName.method`, а также реализован примитивный поиск Java-методов через regex.
2. **Векторный движок (`vector_store.py`)**: Хранит эмбеддинги чанков в ChromaDB. Для создания эмбеддингов используется `sentence-transformers` (модель `paraphrase-multilingual-MiniLM-L12-v2`). Также движок включает классический алгоритм **BM25 (`bm25.py`)** без внешних зависимостей. BM25 нужен для точного лексического поиска (keyword search) по именам переменных и функциям, что отлично дополняет семантический векторный поиск, образуя мощный гибридный поиск.
3. **RAG Pipeline (`rag_pipeline.py` & `index.py`)**: CLI для индексации и логика генерации ответов. Опционально подключается к Ollama (`mistral:7b`) для выдачи ответов на естественном языке.
4. **UI & Docker (`app.py`, `Dockerfile`)**: Веб-интерфейс на базе Streamlit с подсветкой кода `st.code()` и настройки для запуска через Docker Compose.

## Обоснование стратегии чанкования
Стратегия "1 чанк = 1 функция/класс" (с отдельным выделением методов) идеально подходит для RAG-систем поиска по коду, так как функции и методы являются естественными логическими границами программы. Поиск по смыслу работает точнее, когда контекст не разрывается на середине функции (что бывает при случайном чанкинге по количеству токенов вроде 500 токенов). AST-парсер извлекает ровно один смысловой блок кода.

## Обоснование параметров Гибридного поиска (Выбор Alpha)
В системе реализован гибридный поиск (BM25 + Косинусное сходство). Баланс между текстовым (keyword) и векторным (semantic) поиском настраивается параметром **Alpha** (от 0.0 до 1.0).
Для поиска оптимального значения было проведено эмпирическое исследование на трёх различных кодовых базах:
1. `FastAPI` (внутренний тестовый проект)
2. `Requests` (psf/requests)
3. `Click` (pallets/click)

В качестве метрики использовался **Mean Precision@5** на наборе из 31 вопроса (как на русском, так и на английском языках, включая вопросы на точное совпадение имён классов/переменных и семантические вопросы). 

**Результаты эксперимента:**
- При `Alpha = 0.00` (Только BM25 текстовый поиск): **Precision@5 = 0.3065**
- При `Alpha = 1.00` (Только векторный семантический поиск): **Precision@5 = 0.6505**
- При `Alpha = 0.75` (Смешанный гибридный поиск): **Precision@5 = 0.7258**

**Вывод:** Наш эксперимент доказывает эффективность гибридного подхода. Чистый векторный поиск работает отлично для русскоязычных запросов к английскому коду, но проседает на специфичных терминах. Подмешивание текстового алгоритма BM25 (на 25%) даёт значительный прирост к итоговой релевантности. Рекомендуемое значение для приложения: **Alpha = 0.75**.

## Инструкции по запуску

### Локальный запуск
1. Создайте виртуальное окружение и активируйте его:
   ```bash
   python3.12 -m venv venv
   source venv/bin/activate
   ```
2. Установите зависимости (исключительно Python-модули):
   ```bash
   pip install -r requirements.txt
   ```
3. Установите **Ollama** локально для аппаратного ускорения:
   - Скачайте Ollama с официального сайта [ollama.com](https://ollama.com/) и установите.
   - Скачайте модель mistral:
   ```bash
   ollama pull mistral:7b
   ```
   *Запуск Ollama нативно вне докера позволяет использовать GPU/Metal для сверхбыстрых ответов.*
3. Проиндексируйте исходный код:
   ```bash
   python index.py <путь_к_папке>
   # Пример: python index.py docs/dataset_case3_v1.0_fix/gymhero
   ```
4. Запустите интерфейс:
   ```bash
   streamlit run app.py
   ```

### Docker Compose
Запустите систему одной командой:
```bash
docker-compose up --build
```
Интерфейс будет доступен по адресу http://localhost:8501.
Для использования LLM в Docker потребуется загрузить модель mistral:
```bash
docker compose exec ollama ollama pull mistral:7b
```

### Вариант 3. Использование в качестве MCP-сервера
Система поддерживает интеграцию с локальными AI-агентами (например, Claude Desktop или Cursor) через протокол MCP. Это позволяет вашему агенту не читать весь код самостоятельно (что тратит токены), а поручить поиск и агрегацию серверу CodeLens.

**Инструкция по настройке для Claude Desktop:**
1. Установите зависимости, включая пакет `mcp`.
2. Добавьте в конфигурационный файл Claude Desktop (обычно `~/Library/Application Support/Claude/claude_desktop_config.json`) следующий блок:
```json
{
  "mcpServers": {
    "CodeLens": {
      "command": "/Абсолютный/Путь/К/Этому/Проекту/venv/bin/python",
      "args": [
        "-m",
        "mcp_server"
      ],
      "cwd": "/Абсолютный/Путь/К/Этому/Проекту"
    }
  }
}
```
После перезапуска Claude у вас появятся инструменты: `index_codebase`, `search_code`, и `ask_codebase`!

## Примеры запросов и ответов
Ниже приведены 5 примеров тестовых запросов и краткое описание ожидаемых ответов от системы RAG:

1. **Запрос:** `как в проекте создаётся токен доступа и какой срок его жизни?`
   **Ответ:** Система находит функцию `create_access_token` в файле аутентификации. LLM отвечает: *"Токен создаётся функцией `create_access_token` с использованием алгоритма HS256. Срок жизни по умолчанию задан в `ACCESS_TOKEN_EXPIRE_MINUTES`."*

2. **Запрос:** `how does the project create a new account?`
   **Ответ:** Извлекается эндпоинт регистрации и/или CRUD функция создания пользователя. LLM отвечает: *"A new account is created via the `create_user` function, which accepts a `UserCreate` schema and hashes the password before saving it to the database."*

3. **Запрос:** `где в проекте проверяется, является ли пользователь суперпользователем?`
   **Ответ:** Находится функция/свойство `is_superuser`. LLM отвечает: *"Проверка на суперпользователя осуществляется в функции `get_current_active_superuser`, которая проверяет флаг `is_superuser` у текущего пользователя."*

4. **Запрос:** `how does the project prevent adding the same training unit to a plan twice?`
   **Ответ:** Система находит блок проверки на существование связи между юнитом и планом. LLM отвечает: *"The project prevents adding the same unit twice by checking if `unit_id` already exists in `plan.units` before appending."*

5. **Запрос:** `как настроена поддержка WebSocket-соединений в проекте?`
   **Ответ:** Поскольку код FastAPI не содержит импортов `WebSocket`, система не находит релевантных фрагментов. LLM честно отвечает: *"В предоставленном коде нет информации о настройке WebSocket-соединений. Проект использует только стандартные HTTP-эндпоинты."*
